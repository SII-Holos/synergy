Edit the Lattice Pathway. Use the intent that matches the current Lattice phase:
- initial_planning: call with `steps` only to create the ordered Pathway. Omit `id` for new steps; `id` means updating an existing non-terminal step.
- step_blueprinting: call with `bindCurrentBlueprint` only after writing the Blueprint note for the current step.
- result_analysis: call with `recordResult` for the terminal step result. You may also include `steps` only when adding or reshaping future recovery work.

Do not send empty placeholder objects. Omit unused fields entirely. Empty placeholder `bindCurrentBlueprint.noteID`, `recordResult.stepID`, or `steps:[]` are ignored by the tool and do not count as intents.

Intent details:
- steps: replace the ordered list of non-terminal steps (create, update, reorder, split, merge, or drop future steps). Terminal steps (completed/failed/cancelled) are preserved and immutable. Provide each step's title, objective, and optional acceptanceCriteria/assumptions. Reference an existing non-terminal step by id to update it; omit id to create a new one. A recovery step may set addressesFailedStepIDs to reference the failed step(s) it replaces.
- bindCurrentBlueprint: bind a Blueprint note to the current step (use in step_blueprinting after writing the Blueprint note).
- recordResult: attach a result summary to a terminal step (use in result_analysis).

The backend advances the phase automatically after your patch. You cannot restructure the Pathway or modify the running step during blueprint_execution, and you cannot alter terminal steps.
