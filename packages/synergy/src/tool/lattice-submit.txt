Submit exactly one semantic action for the current parent Lattice state. Valid state/action pairs are: submit_requirements in clarifying; submit_pathway in planning; submit_pathway_review in reviewing_pathway; submit_blueprint in blueprinting; submit_blueprint_review in reviewing_blueprint; and approve_execution in awaiting_execution. The strict action contracts are: submit_requirements uses goal, successCriteria, and optional constraints/nonGoals/assumptions; submit_pathway uses reason; submit_pathway_review uses reason; submit_blueprint uses blueprintID; submit_blueprint_review uses reason; approve_execution uses reason and only follows explicit user approval. Never mix fields from different actions.

Result semantics are operational instructions. LATTICE_SUBMIT_INVALID_ARGUMENTS means no action was persisted: correct only the stated fields and retry the same current-state action once. LATTICE_ACTION_WRONG_STATE, LATTICE_RUN_NOT_ACTIVE, LATTICE_ACTION_CONFLICT, LATTICE_BLUEPRINT_REVIEW_STALE, or LATTICE_RUN_NOT_FOUND means no action was persisted and the result is not retryable: follow the stated current owner, do not bypass Lattice with other tools, and do not begin another state or future Pathway Step. LATTICE_BLUEPRINT_UNAVAILABLE is retryable only when the result explicitly says to repair the current Blueprint artifact and retry once. LATTICE_ACTION_QUEUED means the semantic intent is durable but state has not changed synchronously. LATTICE_ACTION_ALREADY_QUEUED means an exact retry matched the existing durable intent. After either successful result, end the assistant turn immediately; do not call tools, poll, resubmit, create next-state artifacts, or execute future work. Only the host controller validates the intent, advances state, and delivers the next responsibility after the turn ends.
