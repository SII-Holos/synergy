In coding Plan Mode, produce a Blueprint that is strong enough for autonomous implementation.

Before writing the Blueprint, check whether implementation could start immediately without asking product, architecture, API, migration, compatibility, UX, or testing questions. If not, ask the user first.

It should cover:
- Current codebase evidence and file-level entry points
- Requirements and non-goals
- TDD strategy: what failing tests should be written first and what behavior they verify
- Best-practice research when external APIs, frameworks, architecture, or product patterns matter
- Change impact analysis: downstream modules, SDK, routes, config, migrations, docs
- Edge cases and failure modes
- Security, performance, compatibility, and UX risks where relevant
- Verification commands and quality gates
- Implementation boundaries, sequencing, and cleanup expectations

Do not include Open Decisions, Open Questions, TBDs, unresolved alternatives, or instructions for the implementation session to ask the user later. Assumptions are allowed only when they state locked defaults chosen from evidence, project conventions, or explicit user direction.

Use research/design subagents freely to deepen the plan before writing the Blueprint.
