# Skills, Commands, and Plugins

## Choosing between Skills, Commands, and Plugins

| Need | Use | Why |
|---|---|---|
| Teach Synergy a domain-specific workflow or knowledge | Skill | Injected into agent context; supports progressive disclosure |
| Quick reusable prompt template (e.g., "/review") | Command | Lightweight; no context cost until invoked |
| Add new tools or custom provider auth | Plugin | Runs code; can define tools and auth loaders |
| Inject reference documentation into context | Skill | Use `references/` for large docs |
| One-off prompt you run frequently | Command | Fastest to create; just a .md file |

**Skill vs Command:** Skills are always available in context (metadata) and loaded on demand (body). Commands are only triggered explicitly by the user. If you want the agent to proactively use domain knowledge, use a Skill. If you just want a shortcut prompt, use a Command.

**Skill vs Plugin:** Skills provide knowledge (text). Plugins provide capabilities (code). If you need new tools or API integrations, use a Plugin. If you need the agent to follow a specific procedure, use a Skill.

**Command vs Plugin:** Commands are prompt templates. Plugins are executable code. If you need to run custom logic, use a Plugin. If you just need a canned prompt, use a Command.

## Skills

Skills extend Synergy with specialized knowledge, workflows, and tool integrations. They are modular markdown files that teach Synergy domain-specific procedures.

### Skill file format

Every skill is a `SKILL.md` file (or `Skill.md` 鈥?both casings are discovered) with YAML frontmatter. Plugin skills may alternatively supply content via `content.txt` or `content.md` where applicable.

```markdown
---
name: my-skill
description: "Description of when to use this skill. Include triggers and use cases."
---

# My Skill

Instructions and guidance for using the skill.
```

**Required frontmatter:** `name`, `description`.

The `description` is critical 鈥?it's the only text Synergy sees when deciding whether to load the skill. Be comprehensive about what the skill does and when to trigger it.

The body is only loaded AFTER the skill triggers, so "When to Use" sections in the body are wasted.

### Skill directories

Skills are discovered from these directories:

| Scope | Path pattern |
|---|---|
| Project | `<project>/.synergy/{skill,skills}/**/SKILL.md` |
| Global | `~/.synergy/config/{skill,skills}/**/SKILL.md` |
| Claude compat (project) | `<project>/.claude/skills/**/SKILL.md` |
| Claude compat (global) | `~/.claude/skills/**/SKILL.md` |
| Codex compat (project) | `<project>/.codex/skills/**/SKILL.md` |
| Codex compat (global) | `~/.codex/skills/**/SKILL.md` |
| Agents compat (project) | `<project>/.agents/skills/**/SKILL.md` |
| Agents compat (global) | `~/.agents/skills/**/SKILL.md` |
| OpenClaw compat (global) | `~/.openclaw/skills/**/SKILL.md` |
| Legacy project root | `<project>/skills/**/SKILL.md` |

Skill entry files support `SKILL.md` and `Skill.md` (both casings). Plugin skills may supply content via `content.txt` or `content.md` as an alternative to `SKILL.md`.

### Progressive disclosure

Skills use three-level loading to manage context:

1. **Metadata** (name + description) 鈥?always in context (~100 words)
2. **SKILL.md body** 鈥?loaded when skill triggers (keep under 500 lines)
3. **Bundled resources** 鈥?loaded by agent as needed (unlimited)

When SKILL.md approaches 500 lines, split detailed content into `references/` files and reference them clearly:
```markdown
For API details, see references/api-docs.md.
For conversion rules, see references/format-rules.md.
```

### Skill resources

- **`references/`** 鈥?documentation loaded into context as needed. Keep SKILL.md lean.
- **`scripts/`** 鈥?executable code (Python/Bash) for deterministic tasks. Token-efficient.
- **`assets/`** 鈥?files used in output (templates, images). Not loaded into context.

### Reload

```
runtime_reload(target: "skill")
```

File watcher also auto-detects SKILL.md changes.

## Commands

Custom commands are reusable prompt templates triggered via `/command-name` in TUI/Web.

### Command file format

Create `.synergy/command/<name>.md` (project) or `~/.synergy/config/command/<name>.md` (global):

```markdown
---
description: "Run code review on staged changes"
agent: "synergy"
model: "anthropic/claude-sonnet-4-5"
---

Review the staged git changes. Focus on:
- Security issues
- Performance concerns
- Code style violations

Provide a summary with severity ratings.
```

**Frontmatter field reference:**

| Field | Type | Default | Description |
|---|---|---|---|
| `description` | string | 鈥?| Shown in command list UI. Required. |
| `agent` | string | default agent | Which agent handles this command |
| `model` | string | agent's model | Model override for this command |

**Body** = the prompt template that gets injected.

### Nested command example

`.synergy/command/review/security.md` 鈫?`/review/security`

```markdown
---
description: "Run security audit on changed files"
agent: "developer"
model: "anthropic/claude-sonnet-4-5"
---

Perform a security audit on the staged git changes. Check for:
- SQL injection and XSS vulnerabilities
- Hardcoded secrets and credentials
- Insecure deserialization
- Missing input validation

Output a severity-ranked finding list.
```

### Command directories

| Scope | Path pattern |
|---|---|
| Project | `<project>/.synergy/{command,commands}/**/*.md` |
| Global | `~/.synergy/config/{command,commands}/**/*.md` |

Nested commands: `.synergy/command/review/security.md` 鈫?`/review/security`

### Reload

```
runtime_reload(target: "command")
```

## Plugins

Plugins export one `definePlugin()` definition. plugin-kit generates the installable manifest and bundles; source projects must not contain `plugin.json`.

### Plugin config

Configured plugin specs live in the Plugins domain:

| Scope | Path |
|---|---|
| Project | `<project>/.synergy/synergy.d/50-plugins.jsonc` |
| Global | `~/.synergy/config/synergy.d/50-plugins.jsonc` |

```jsonc
{
  "plugin": ["file:///absolute/path/to/my-plugin"],
  "pluginRuntimePolicy": {
    "limits": {
      "startupTimeoutMs": 15000,
      "toolInvocationTimeoutMs": 120000,
      "hostServiceRequestTimeoutMs": 120000,
      "taskRunTimeoutMs": 120000,
      "shutdownGraceMs": 1500,
      "heartbeatIntervalMs": 5000
    }
  }
}
```

A built project directory resolves to `dist/plugin.json`. Supported specs also include built archives, npm packages, git specs, URLs, official/local registry artifacts, and built-ins. External runtime mode is always `process`; trusted built-ins may use `inProcess`.

### Plugin API

```ts
import z from "zod"
import { capability, definePlugin, operation } from "@ericsanchezok/synergy-plugin"

export default definePlugin({
  id: "my-metrics",
  version: "1.0.0",
  description: "Metrics integration",
  capabilities: [capability("workspace.read")],
  contributions: [
    operation({
      id: "metrics.get",
      type: "query",
      input: z.object({ service: z.string() }),
      output: z.object({ value: z.number() }),
      async handler({ service }, context) {
        context.log.info("query metrics", { service })
        return { value: 0 }
      },
    }),
  ],
})
```

Build and register:

```bash
bunx @ericsanchezok/synergy-plugin-kit create my-plugin
cd my-plugin
bun install
bun run build
bun run validate
bun run pack
synergy plugin add file:///absolute/path/to/my-plugin
```

### Debugging and reload

- Check plugin status, contribution health, runtime logs, manifest generation, and approval in `synergy plugin info <id>` and `synergy plugin runtime ...`.
- Build failures must leave the prior dev generation active.
- Runtime startup rejects API/protocol/generation/handler mismatches.
- A plugin must be enabled for the active Scope before its operations and UI are available.
- Reload configured contributions with `runtime_reload(target: "plugin")`; this also refreshes plugin tools, agents, skills, and MCP declarations.

### Reload

```
runtime_reload(target: "plugin")
```

Plugin reload cascades to the tool registry.

## Custom Tools (lightweight, no SDK)

For simple custom tools that do not need the full plugin SDK, place TypeScript/JavaScript files in a `tool/` directory:

| Scope | Path pattern |
|---|---|
| Project | `<project>/.synergy/tool/*.{ts,js}` |
| Global | `~/.synergy/config/tool/*.{ts,js}` |

Each file exports one or more tool definitions using the internal tool API. Unlike plugins, these are scanned directly from the tool directories 鈥?not configured via `50-plugins.jsonc`.

## Troubleshooting

### Skills

- **Skill not triggering** 鈥?check `description` in frontmatter is comprehensive; description is the ONLY text Synergy uses to decide whether to load the skill; vague descriptions = never loaded
- **Skill content not loading** 鈥?verify the file is named `SKILL.md` or `Skill.md`; check it's in a recognized skill directory; try `runtime_reload(target: "skill")`
- **References not found** 鈥?reference paths in the skill body are relative to the SKILL.md file's directory; use `references/filename.md` not absolute paths

### Commands

- **Command not appearing** 鈥?verify file is `.md` in a `command/` or `commands/` directory; check frontmatter has `description`; try `runtime_reload(target: "command")`
- **Command using wrong agent** 鈥?check `agent` field in frontmatter; if omitted, uses the default agent

### Plugins

- **Plugin tools not appearing** 鈥?reload with `runtime_reload(target: "plugin")`; check server logs for load errors; verify the tool schema is valid JSON Schema
- **Plugin MCP tools not appearing** 鈥?reload with `runtime_reload(target: "plugin")` after plugin changes, or `runtime_reload(target: "mcp")` after MCP-only changes. Lazy plugin MCP servers must be connected before their tools appear in agent tool discovery.
- **Plugin auth not working** 鈥?auth loaders must return a valid credential format; check credential store for the resulting entry
