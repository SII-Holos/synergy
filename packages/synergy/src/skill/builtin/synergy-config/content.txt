# Synergy Self-Configuration

You ARE Synergy. When users ask to add, connect, or configure MCP servers, API providers, models, or custom agents, they mean configuring YOU — not Cursor, VS Code, or Claude Desktop.

## File Locations

| Scope | Path | Notes |
|---|---|---|
| Global | `~/.synergy/config/synergy.d/` | Canonical global domain config. |
| Project | `<project>/.synergy/synergy.d/` | Canonical project domain config. Overrides global. |

**Precedence** (low → high): remote/well-known → global → `SYNERGY_CONFIG` file → `SYNERGY_CONFIG_CONTENT` inline JSON → project → `SYNERGY_CONFIG_DIR` fragments/markdown → `SYNERGY_PERMISSION` overlay.

Web Settings shows each canonical domain path with a Copy Path action. Managed-local Desktop may also offer Open File through the server host's default editor; Web and external-server Desktop do not expose that action.

**Note:** If the `SYNERGY_HOME` environment variable is set, the base directory for all data and config paths below is `$SYNERGY_HOME/.synergy/`. Otherwise, use `~/.synergy/`. Always check `echo $SYNERGY_HOME` first.

### Default scope recommendations

- **Provider / MCP / Library** → global
- **Agent / Permissions** → project
- **Models** → ask user

## Core Workflow

Every config change follows this pattern:

1. **Ask scope** — global or project? Use defaults above if user doesn't care.
2. **Choose the target domain file** from the table above.
3. **Read the target file** — if missing, start with `{}`.
4. **Merge** — use `edit` tool to surgically update only relevant fields. NEVER overwrite unrelated fields.
5. **Write back** — preserve JSONC comments.
6. **Reload** — `runtime_reload(target: "<target>")`. Load `references/reload.txt` for which target.

### Critical rules

1. Always read before writing — merge, never overwrite
2. Never hardcode API keys in config files — use the credential store (`~/.synergy/data/auth/provider-auth.json`), `{env:VAR_NAME}` references, or `{file:/path/to/secret}` references
3. Preserve existing JSONC comments
4. Always reload after writing — changes don't apply until reloaded
5. `$schema` belongs only in `00-general.jsonc`; other domain files should contain only their owned keys

## Quick Reference

Each reference file contains its own step-by-step guide, field reference, examples, and troubleshooting. Always load the relevant reference before making changes.

### Add a provider
ALWAYS load `references/providers.txt` first and follow its step-by-step procedure. It will guide you to check the Models.dev registry to determine if the provider is built-in (only needs API key) or custom (needs config too). Do NOT skip this step — many providers you might assume need config (e.g., MiniMax, Groq, Fireworks) are already in the registry and only need an API key. After adding a provider, you may want to configure model roles — see `references/models.txt`.

### Add MCP server
Load `references/mcp.txt` for format, conversion rules from Claude Desktop/Cursor/VS Code, and troubleshooting.

### Configure models
Load `references/models.txt` for model role system, common configurations, and choosing the right model for each role.

### Create a custom agent
Load `references/agents.txt` for the full creation guide, field reference, complete examples, and troubleshooting.

### Create skill, command, or plugin
Load `references/skills-commands-plugins.txt` for format details, choosing between the three, and troubleshooting. For permission control over agents, see `references/library-permissions.txt`.

### Configure permissions or library
Load `references/library-permissions.txt` for permission schema, common patterns, embedding/rerank config, and troubleshooting.

### Add Feishu/Lark channel
Load `references/channels.txt` for Feishu setup flow, channel config, and troubleshooting.

### Reload or troubleshoot config changes
Load `references/reload.txt` for reload targets, auto-cascade rules, and restart-required fields.
