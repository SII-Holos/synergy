## File Locations

| Scope | Path | Notes |
|---|---|---|
| Global | `~/.synergy/config/synergy.d/` | Canonical global domain config. |
| Project | `<project>/.synergy/synergy.d/` | Canonical project domain config. Overrides global. |

**Precedence** (low ‚Üí high): remote/well-known ‚Üí global ‚Üí project ‚Üí `SYNERGY_CONFIG_CONTENT` env var.

Web Settings uses these same canonical domains. Common fields are editable in forms, and complex domains can be opened from Settings with the system default editor.

**Note:** If the `SYNERGY_HOME` environment variable is set, the base directory for all data and config paths below is `$SYNERGY_HOME/.synergy/` instead of `~/.synergy/`. Always check `echo $SYNERGY_HOME` first ‚Äî if empty, use `~/.synergy/` as normal.

## Core Workflow

Every config change follows this pattern:

1. **Ask scope** ‚Äî global or project? Use defaults above if user doesn't care.
2. **Choose the target domain file** from the table above.
3. **Read the target file** ‚Äî if missing, start with `{}`.
4. **Merge** ‚Äî use `edit` tool to surgically update only relevant fields. NEVER overwrite unrelated fields.
5. **Write back** ‚Äî preserve JSONC comments.
6. **Reload** ‚Äî `runtime_reload(target: "<target>")`. Load `references/reload.txt` for which target.

### Critical rules

1. Always read before writing ‚Äî merge, never overwrite
2. Never hardcode API keys in config files ‚Äî use the credential store (`~/.synergy/data/auth/api-key.json`) or `{env:VAR_NAME}` references
3. Preserve existing JSONC comments
4. Always reload after writing ‚Äî changes don't apply until reloaded
5. `$schema` belongs only in `00-general.jsonc`; other domain files should contain only their owned keys

## Quick Reference

Each reference file contains its own step-by-step guide, field reference, examples, and troubleshooting. Always load the relevant reference before making changes.

### Add a provider
ALWAYS load `references/providers.txt` first and follow its step-by-step procedure. It will guide you to check the Models.dev registry to determine if the provider is built-in (only needs API key) or custom (needs config too). Do NOT skip this step ‚Äî many providers you might assume need config (e.g., MiniMax, Groq, Fireworks) are already in the registry and only need an API key. After adding a provider, you may want to configure model roles ‚Äî see `references/models.txt`.

### Add MCP server
Load `references/mcp.txt` for format, conversion rules from Claude Desktop/Cursor/VS Code, and troubleshooting.

### Configure models
Load `references/models.txt` for model role system, common configurations, and choosing the right model for each role.

### Create a custom agent
Load `references/agents.txt` for the full creation guide, field reference, complete examples, and troubleshooting.

### Create skill, command, or plugin
Load `references/skills-commands-plugins.txt` for format details, choosing between the three, and troubleshooting. For permission control over agents, see `references/library-permissions.txt`.

### Configure permissions or library
Load `references/library-permissions.txt` for permission schema, common patterns, embedding/rerank config, and troubleshooting.

### Add Feishu/Lark channel
Load `references/channels.txt` for Feishu setup flow, channel config, and troubleshooting.

### Reload or troubleshoot config changes
Load `references/reload.txt` for reload targets, auto-cascade rules, and restart-required fields.
