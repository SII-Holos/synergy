# Plugin UI Contributions

Plugins can extend the Web workbench with versioned manifest surfaces. The host owns placement, lifecycle, shared state, accessibility, and error handling; the plugin supplies declared metadata, optional packaged assets, and named Solid exports.

## Permission and API Version

Every `contributes.ui` block requires:

```jsonc
{
  "permissions": { "ui": true },
}
```

Solid-backed surfaces also require a built JavaScript entry and minimum UI API version:

```jsonc
{
  "contributes": {
    "ui": {
      "entry": "./ui/index.js",
      "minUIApiVersion": "3.0",
    },
  },
}
```

The current Web host UI API is `3.0` and compatibility requires the same major version. Use the version generated by the installed plugin kit rather than copying a stale example.

The build must externalize `solid-js`, `solid-js/web`, `solid-js/store`, and supported Solid JSX/H runtime paths. Synergy rewrites those imports to the host's shared Solid runtime and rejects bundles that embed their own Solid copy or import unsupported subpaths.

## Surface Catalog

| Surface           | Purpose                              | Solid entry required                             |
| ----------------- | ------------------------------------ | ------------------------------------------------ |
| `toolRenderers`   | Render a declared plugin tool        | No, if every renderer has a declarative fallback |
| `partRenderers`   | Render a custom message-part type    | Yes                                              |
| `workbenchPanels` | Add side or bottom workbench content | Yes                                              |
| `navigation`      | Add a sidebar or page destination    | Yes                                              |
| `settings`        | Add a settings section               | No, when `formSchema` is supplied                |
| `messageSlots`    | Add content at a named message slot  | Yes                                              |
| `composerSlots`   | Add content around the composer      | Yes                                              |
| `commands`        | Register a Web UI command            | Yes                                              |
| `themes`          | Register a structured JSON theme     | No                                               |
| `icons`           | Register packaged SVG icons          | No                                               |

Surface IDs are namespaced by the host as `<plugin-id>:<surface-id>`. Navigation routes are generated under `/plugins/<plugin-id>/<navigation-id>`.

## Tool Renderers

Declare the tool name and named export:

```jsonc
{
  "toolRenderers": [
    {
      "tool": "lookup",
      "exportName": "LookupRenderer",
      "fallback": {
        "icon": "search",
        "title": "Lookup",
        "subtitleTemplate": "{{id}}",
      },
    },
  ],
}
```

The fallback provides a useful host-rendered card if no JS entry is present or while the custom renderer is unavailable. A custom renderer changes presentation only; it does not change tool schemas, results, permissions, or transcript semantics.

## Workbench Panels

A panel declares:

- `surface`: `side` or `bottom`
- `cardinality`: `exclusive`, `singleton`, or `multi`
- label, icon, order, and named export
- optional `requiresSession`

Use panels for persistent task-adjacent work such as a preview, inspector, or domain workspace. Let the workbench control tabs, focus, close behavior, session binding, and restoration. Do not build a second application shell inside the panel.

## Navigation

Navigation contributions use `placement: "sidebar"` or `"page"`. The host assigns the route and namespaces the registration. Use them for a durable plugin destination, not as a shortcut for transient tool output.

## Settings

Settings can be declarative:

```jsonc
{
  "settings": [
    {
      "id": "connection",
      "label": "Connection",
      "group": "Plugins",
      "formSchema": {
        "endpoint": { "type": "string", "label": "Endpoint" },
      },
    },
  ],
}
```

or a named Solid export. Prefer a form schema for ordinary plugin configuration: it follows host layout and reduces executable UI. Use a component when the interaction cannot be represented declaratively.

Credentials belong in the plugin auth store, not ordinary settings fields or browser-local storage.

## Message and Composer Slots

Message slots use registered dotted slot names. Composer slots are restricted to:

- `composer.above`
- `composer.below`
- `composer.toolbar.left`
- `composer.toolbar.right`
- `composer.add-menu`
- `composer.start-option`

Slots should remain small and contextual. A surface that needs its own navigation, long-lived state, or substantial layout belongs in a navigation page or workbench panel.

## Commands, Themes, and Icons

UI commands expose a label, optional description/icon, and named command export. They are registered in the host command registry under the plugin namespace.

Themes are packaged JSON assets with complete light and dark seed palettes. The host validates the theme, generates every canonical color token through the shared resolver, namespaces the selected theme ID, and applies the same runtime path used by the built-in Synergy theme.

```jsonc
{
  "themes": [{ "id": "ocean", "label": "Ocean", "path": "./themes/ocean.json" }],
}
```

The referenced JSON uses this shape:

```jsonc
{
  "name": "Ocean",
  "id": "ocean",
  "light": {
    "seeds": {
      "neutral": "#64748B",
      "primary": "#0284C7",
      "success": "#16A34A",
      "warning": "#D97706",
      "error": "#DC2626",
      "info": "#0EA5E9",
      "interactive": "#0284C7",
      "diffAdd": "#16A34A",
      "diffDelete": "#DC2626",
    },
  },
  "dark": {
    "seeds": {
      "neutral": "#94A3B8",
      "primary": "#38BDF8",
      "success": "#4ADE80",
      "warning": "#FBBF24",
      "error": "#F87171",
      "info": "#38BDF8",
      "interactive": "#38BDF8",
      "diffAdd": "#4ADE80",
      "diffDelete": "#F87171",
    },
  },
}
```

Each variant may add `overrides` keyed only by canonical theme token names. Hex values and `var(--token)` references are accepted; unknown keys and arbitrary CSS are rejected. Preserve the Web polarity rule and verify common text/surface pairs at WCAG AA contrast. Use the plugin-kit `theme-icon` template as the current schema example. CSS-based theme contributors should follow [the structured-theme migration](../migrations/plugin-theme-json.md).

Icons are packaged SVG assets loaded into the plugin icon registry. Treat SVG as code-bearing input: ship static reviewed assets, avoid remote fetches, and keep names plugin-specific.

## Loading and Failure Behavior

The server exposes enabled contributions and versioned asset URLs. The Web host fetches each JS bundle, verifies the UI API major, rejects embedded or unsupported Solid runtimes, imports the requested named export, and registers a disposer for every surface.

One missing export or invalid surface is reported against that plugin. Reloading disposes the plugin's old registrations before adding the new set.

## Authoring Rules

- Use the plugin-kit templates for the current build and shared-runtime setup.
- Declare only surfaces the product actually needs.
- Provide a declarative renderer fallback where a custom tool card is optional.
- Keep tool results meaningful without custom UI.
- Use host navigation and workbench contracts instead of recreating them.
- Keep keyboard, focus, loading, empty, error, and narrow-layout states usable.
- Package every referenced JS, JSON, CSS, and SVG asset; never rely on source paths after build.
- Validate exports and build output before installation.
