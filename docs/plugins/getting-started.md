# Plugin Getting Started

Use Bun and the published Plugin Development Kit. Clone the Synergy repository only when changing the platform itself.

## Create a Project

```bash
bunx @ericsanchezok/synergy-plugin-kit create my-plugin --template tool-ui
cd my-plugin
bun install
```

Current templates are:

- `tool-ui`
- `workbench-panel`
- `navigation`
- `api-connector`
- `theme-icon`

The generated project pins compatible plugin SDK and kit packages and provides scripts for development, validation, build, test, pack, signing, and marketplace publishing.

## Runtime Descriptor

`src/index.ts` exports one descriptor object:

```ts
import type { PluginDescriptor, PluginHooks, PluginInput } from "@ericsanchezok/synergy-plugin"
import { tool } from "@ericsanchezok/synergy-plugin/tool"

const greet = tool({
  description: "Greet a user",
  args: {
    name: tool.schema.string(),
  },
  async execute(args) {
    return { output: `Hello, ${args.name}` }
  },
})

export const plugin: PluginDescriptor = {
  id: "my-plugin",
  name: "My Plugin",
  async init(_input: PluginInput): Promise<PluginHooks> {
    return {
      tool: { greet },
    }
  },
}

export default plugin
```

Use `tool` as the hook key. The descriptor ID must equal `plugin.json.name`.

`PluginInput` provides the generated client, current Scope, workspace directory, server URL, guarded shell helper, plugin-scoped config/auth/cache accessors, and the absolute plugin package directory. These services are direct objects for in-process plugins and host-bridge proxies for isolated plugins.

## Manifest

Declare the runtime tool and its minimum permissions before installation imports it:

```jsonc
{
  "name": "my-plugin",
  "version": "0.1.0",
  "description": "Example plugin",
  "engines": {
    "synergy": ">=2.4.3",
  },
  "main": "./src/index.ts",
  "permissions": {
    "tools": {
      "filesystem": "none",
      "network": false,
      "shell": false,
      "mcp": "none",
    },
  },
  "contributes": {
    "tools": [
      {
        "name": "greet",
        "title": "Greet",
        "description": "Greet a user",
        "capabilities": {
          "filesystem": "none",
          "network": false,
          "shell": false,
        },
      },
    ],
  },
}
```

Use the compatibility range generated by your installed kit; do not copy the example range forward when a newer kit changes it.

## Develop and Validate

```bash
synergy-plugin dev
synergy-plugin validate --runtime-discovery
synergy-plugin test
```

`dev` validates the manifest, displays its risk/permission summary, and watches `src/`. It does not silently install the plugin into a running Synergy instance.

Install or refresh a local source project explicitly:

```bash
synergy plugin add file:///absolute/path/to/my-plugin
```

Runtime discovery imports the descriptor, calls `init()`, and compares returned runtime tools with `contributes.tools`. Undeclared runtime tools fail validation because the host cannot approve their capabilities safely.

## Build and Package

```bash
synergy-plugin build
synergy-plugin pack
```

Build creates `dist/` with:

```text
plugin.json
plugin.normalized.json
runtime/index.js
permissions.summary.json
integrity.json
package.json              # when the source project has one
ui/index.js               # when a compiled UI entry is declared
declared assets
```

The runtime bundle keeps the public Synergy SDK packages external. UI build keeps `solid-js`, `solid-js/web`, `solid-js/store`, and `solid-js/h` external so the host and plugin share one Solid runtime.

`pack` requires the build outputs and creates:

```text
my-plugin-0.1.0.synergy-plugin.tgz
```

The archive is the installable artifact. A GitHub source archive is not equivalent.

## Sign and Publish

```bash
synergy-plugin sign my-plugin-0.1.0.synergy-plugin.tgz
synergy-plugin publish-market
```

Signing creates an Ed25519 keypair on first use, keeps the private key in the local plugin-kit key store, and writes a `.sig` containing artifact, manifest, and permission hashes.

`publish-market` performs the supported official flow: validate, build, pack, sign, prepare release assets, update and validate a registry checkout, and open or leave instructions for a registry pull request. See [Marketplace and registry](marketplace.md).

## Install Sources

The runtime resolver accepts:

```bash
synergy plugin add file:///absolute/path/to/plugin
synergy plugin add file:///absolute/path/to/plugin/src/index.ts
synergy plugin add file:///absolute/path/to/plugin.synergy-plugin.tgz
synergy plugin add npm-package-name
synergy plugin add github:owner/repository
```

Local directories and source entries are development inputs. Packed archives, npm, Git, URL, and official marketplace sources are resolved, cached/staged, validated, and recorded in `plugin.lock`.

Inspect and repair installation state with:

```bash
synergy plugin list
synergy plugin info my-plugin
synergy plugin permissions my-plugin
synergy plugin runtime status my-plugin
synergy plugin doctor
synergy plugin doctor --fix
```

Plugin failures are isolated: a failed plugin is disabled with a reason while built-in tools, other plugins, MCP, and sessions continue.

## When a Source Checkout Is Appropriate

Clone Synergy when changing or debugging:

- SDK or manifest types
- plugin-kit commands/templates
- spec resolution, lockfiles, install transactions, or lifecycle
- approval, trust, signature, or runtime isolation policy
- host bridge enforcement
- marketplace registry and install routes
- Web UI contribution loading
- generated SDK/OpenAPI contracts

For ordinary plugin behavior, stay in the standalone plugin project.
